# Collaborative Communication Extension — Implementation Plan

## Context

Internal actors today are deaf. When Actor A posts in `#research`, Actor B doesn't know it happened. `notify()` only checks `watched_entities` (random 30% hash of entity IDs) — no concept of interest-based listening. This extension adds **ears** to internal actors through subscriptions, structured interaction history, and collaboration-aware prompts. No new engines — all modifications to existing AgencyEngine, ActorState, and the World Compiler.

**Key design decisions from discussion:**
- Subscriptions are **LLM-inferred at compile time** (not hardcoded rules)
- **`tagged` mode (default)** for token efficiency — only `intended_for` actors activate
- **`open` mode** opt-in for small worlds where emergence matters
- **Internal-only worlds** are first-class — no external agent required
- **Lead actor** with synthesis deadline produces deliverables
- **Deliverable presets** are YAML config files, not hardcoded types
- **Chat auto-included** when internal agents exist
- All communication goes through services (chat pack) — no direct actor-to-actor channel

```python
    """What this actor is listening to in the world.

    Generated by the compiler at compile time based on actor role
    and service topology. NOT hardcoded — LLM infers appropriate
    subscriptions from world context.

    Replaces the flat string list in recent_interactions.
    Provides structured context for the LLM prompt.

    actor_role: str                    # "researcher-A" | "customer" | "supervisor"
    action: str                        # "slack_post_message" | "email_send" etc.

    source: Literal["self", "observed", "notified"]  # how this actor learned about it
    event_id: str                      # reference to original WorldEvent
    reply_to: str | None = None        # event_id this was a reply to
    channel: str | None = None         # e.g. "#research" for chat messages
```

#### 2b. New method: `_matches_subscription()`

```python
def _matches_subscription(self, event: WorldEvent, sub: Subscription) -> bool:
    """Check if a committed event matches an actor's subscription."""
    # Service must match
    if str(event.service_id) != sub.service_id:
        return False

    # Match filter criteria against event payload and metadata
    for key, value in sub.filter.items():
        if value == "self":
            continue  # resolved at activation time, not here

        # Check event input_data
        if key in event.input_data and event.input_data[key] == value:
            continue

        # Check event metadata
        if key in event.metadata and event.metadata[key] == value:
            continue

        # Check response_body
        if event.response_body and key in event.response_body and event.response_body[key] == value:
            continue

        # No match on this filter key
        return False

    return True
```

#### 2c. New method: `_build_interaction_record()`

```python
def _build_interaction_record(
    self, event: WorldEvent, observer: ActorState, source: str
) -> InteractionRecord:
    """Build a structured interaction record from a WorldEvent."""
    # Extract summary from event
    content = event.input_data.get("content", "")
    text = event.input_data.get("text", "")
    body = event.input_data.get("body", "")
    subject = event.input_data.get("subject", "")
    summary_text = content or text or body or subject or event.action

    # Truncate to reasonable length
    if len(summary_text) > 200:
        summary_text = summary_text[:197] + "..."

    # Get actor role from actor states
    actor_role = ""
    actor_state = self._actor_states.get(event.actor_id)
    if actor_state:
        actor_role = actor_state.role

    return InteractionRecord(
        tick=event.timestamp.tick,
        actor_id=str(event.actor_id),
        actor_role=actor_role,
        action=event.action,
        summary=summary_text,
        source=source,
        event_id=str(event.event_id),
        reply_to=event.input_data.get("reply_to_event_id"),
        channel=event.input_data.get("channel"),
        intended_for=event.input_data.get("intended_for", []),
    )
```

### 7. Subscription Generator (`terrarium/engines/world_compiler/subscription_generator.py`)

**New file.** Uses LLM to infer subscriptions at compile time.

```python
"""Generate actor subscriptions from world context via LLM.

The compiler calls this during the actor compilation phase. The LLM
sees the world description, service topology, actor role/persona,
and generates appropriate subscriptions. NO hardcoded rules.
"""

class SubscriptionGenerator:
    """LLM-based subscription inference."""

    def __init__(self, llm_router: LLMRouter):
        self._llm_router = llm_router

    async def generate_subscriptions(
        self,
        actor_spec: dict,
        world_plan: WorldPlan,
    ) -> list[Subscription]:
        """Generate subscriptions for an actor based on world context.

        The LLM sees:
        - Actor role and persona
        - Available services and their channels/entities
        - Other actors in the world
        - The world's mission and collaboration mode

        Returns a list of Subscription objects.
        """
        # Build prompt with full world context
        prompt = self._build_subscription_prompt(actor_spec, world_plan)

        request = LLMRequest(
            system_prompt="You are the Terrarium World Compiler...",
            user_content=prompt,
            output_schema=SUBSCRIPTION_SCHEMA,
        )
        response = await self._llm_router.route(request, "world_compiler")

        return self._parse_subscriptions(response.content)
```

The LLM prompt includes service topology (which channels exist, which entity types), actor role, and asks: "What should this actor subscribe to?" The compiler validates output against known services.

```python
@app.get("/api/v1/runs/{run_id}/messages")
async def get_run_messages(
    run_id: str,
    channel: str | None = None,
    limit: int = 50,
    offset: int = 0,
):
    """Get communication messages from a run, formatted for chat view."""
    # Query events where action is a communication action
    communication_actions = {
        "slack_post_message", "slack_reply_message",
        "email_send", "reddit_comment", "reddit_submit",
        "twitter_create_tweet", "twitter_reply",
    }

    events = await gateway._app.get_run_events(
        RunId(run_id),
        action_filter=communication_actions,
        limit=limit,
        offset=offset,
    )

    messages = []
    for evt in events:
        msg = {
            "id": str(evt.event_id),
            "tick": evt.timestamp.tick,
            "actor_id": str(evt.actor_id),
            "actor_role": evt.metadata.get("actor_role", ""),
            "channel": evt.input_data.get("channel", ""),
            "content": (
                evt.input_data.get("content")
                or evt.input_data.get("text")
                or evt.input_data.get("body")
                or ""
            ),
            "reply_to": evt.input_data.get("reply_to_event_id"),
            "intended_for": evt.input_data.get("intended_for", []),
            "timestamp": evt.timestamp.wall_time.isoformat() if evt.timestamp.wall_time else "",
        }
        if channel and msg["channel"] != channel:
            continue
        messages.append(msg)

    return {"messages": messages, "count": len(messages)}
```

## Design Principles Verification

| Principle | How We Follow It |
|---|---|
| **No hardcoded logic** | Subscriptions LLM-inferred, not rule-based. Presets are YAML files. Collaboration mode is config. |
| **No cross-engine imports** | AgencyEngine uses protocols only. Subscription generator is in world_compiler, not agency. |
| **All through pipeline** | Communication goes through chat/email packs → 7-step pipeline. No direct actor-to-actor. |
| **Event bus integration** | Subscription matches recorded to ledger. Notifications go through existing notify() flow. |
| **Ledger everything** | New entry types: `SubscriptionMatchEntry`, `CollaborationNotificationEntry`. |
| **Config-driven** | collaboration_mode, batch_threshold, idle_stop, synthesis_buffer = all TOML config. |
| **Frozen Pydantic models** | `Subscription`, `InteractionRecord` are frozen. Mutable fields only on ActorState. |
| **All async** | LLM calls for subscription generation use async `llm_router.route()`. |
| **Tests alongside code** | Unit + integration + live E2E tests included. |

## Gap Fix #1: Scheduled Periodic Checks (Addition 6 from spec)

**What:** Compiler generates periodic `ScheduledAction` for actors in collaborative roles so they proactively check shared state instead of only reacting to notifications.

**Implementation in `subscription_generator.py`:**

```python
async def generate_periodic_checks(
    self, actor_spec: dict, world_plan: WorldPlan
) -> list[ScheduledAction]:
    """Generate periodic check schedules for collaborative actors.

    The LLM decides the interval based on role urgency and collaboration
    density. Higher urgency = more frequent checks.
    """
    # Only for internal actors in collaborative worlds
    if actor_spec.get("type") != "internal":
        return []
    if not world_plan.services:
        return []

    # LLM infers check frequency from world context
    # (same pattern as subscription generation — no hardcoded intervals)
    prompt = self._build_periodic_check_prompt(actor_spec, world_plan)
    response = await self._llm_router.route(LLMRequest(...), "world_compiler")
    intervals = self._parse_check_intervals(response.content)

    # Generate ScheduledAction objects at regular intervals
    actions = []
    for interval in intervals:
        for tick in range(interval["start_tick"], world_plan.max_ticks, interval["every_n_ticks"]):
            actions.append(ScheduledAction(
                logical_time=float(tick),
                action_type="review_shared_state",
                description=interval["description"],
                target_service=interval.get("service"),
                payload={"scope": "subscriptions"},
            ))
    return actions
```

**Integration:** Called during `configure_agency()` in `app.py`. ScheduledActions are set on the actor's state. The existing `_tier1_activation_check()` already handles scheduled actions (line ~254).

## Gap Fix #4: Kickstart for Internal-Only Worlds + Simulation Type Config

**Config in `terrarium.toml`:**

```toml
[simulation]
type = "auto"                      # auto | external_driven | internal_only | mixed
max_ticks = 100
```

`auto` means: if world has external actors → `external_driven`. If all actors are internal → `internal_only`. If both → `mixed`.

**Kickstart mechanism in `SimulationRunner`:**

For internal-only worlds, the simulation needs a trigger event to start the collaboration:

```python
# In SimulationRunner.run(), at the start:
if self._simulation_type == "internal_only":
    # Create kickstart event from world seeds
    kickstart = ActionEnvelope(
        envelope_id=EnvelopeId(f"kickstart-{uuid.uuid4().hex[:8]}"),
        actor_id=ActorId("system"),
        source=ActionSource.ENVIRONMENT,
        action_type="collaboration_start",
        target_service=ServiceId("chat"),
```
